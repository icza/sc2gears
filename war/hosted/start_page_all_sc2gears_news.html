October 7, 2015 - Patch 3.0 - The END of Sc2gears

StarCraft II's Patch 3.0 rendered Sc2gears' parsing engine incapable of doing its work. In short: Sc2gears can no longer parse replays since Patch 3.0.

No further updates will be released for Sc2gears, only Scelight will be maintained. Visit the Scelight home page and start using it.

July 15, 2015 - The Sc2gears Database is shutting down

The Sc2gears Database is shutting down. As of now, no new accounts can be created.

On July 28th, 2015 starting at 12pm Pacific Time, the Sc2gears Database will be unavailable for 24 hours.
On August 4th, 2015 the Sc2gears Database will be permanently shut down at 12:00pm Pacific time.

The reason for shutting down the Sc2gears Database is because it is built using a service of the Google App Engine Platform which will be shut down at the mentioned times.

There will be no more Sc2gears releases. Currently Sc2gears supports only WoL and HotS, and it doesn't support LotV. And this will remain this way.
Most likely when LotV is released, Sc2gears will stop functioning for all replays (including WoL, HotS and LotV) due to replay format changes.

April 14 - Sc2gears Database Migration Status Report

The Sc2gears Database Server is being migrated from M/S datastore to HRD datastore. This migration started on April 11 and finished before midnight of April 11.
There was only a 10-minute read-only period where uploaded replays were rejected. This period took place at around 11 PM (CEST) on April 11. Excluding this short 10-minute period every replay sent to the database were stored properly. If you attempted to store replays around the specified time period, please double check if they got stored properly, if not, simply store them again.

There is a post-task which needs to be completed in order so your previous replays will be accessible. Replays stored after the migration are placed right in the new HRD datastore so they are downloadable. Previous replays (stored before the migration) are listed but not yet downloadable. They will be "back" when the post migration task is completed.

Thank you for your patience. This new HRD datastore provides the highest level of availability and much better performance.

March 8 - Sc2gears Database Issues

On January 30 there was a news stating that the Sc2gears Database service had issues.
These issues arose from a platform service the server used to access the storage layer. It appeared this platform issue had been fixed, but it wasn't.
This means that if you uploaded replays or other files since January 15, there is a slight change they did not get stored (error was reported back in these cases).

Because of this I developed a double-caching storage mechanism which now provides fail-safe file storing operations. If the primary storage is not available, the uploaded file (e.g. replay) will be stored on a secondary storage which is independent from the primary storage. The secondary storage is not affected by the problem mentioned earlier.
The secondary storage is synchronized to the primary automatically in the background.

It is recommended to check if you have missing files. The easiest thing is just to re-upload all your replays and mouse prints that you attempted to store since January 16. The server will recognize if you already have the file and will not store it duplicated.

Again, the error was not in the code of the Sc2gears Database but in the services that was used by it, but I apologize for the inconvenience anyway.

The double-caching storage mechanism will now handle even future errors. The Sc2gears Database operates fail-safe from now on.

January 30 - Parsing Service, is there a need for it?

Replay Site Admins, Attention!

I'm planning to open a parsing service where you could post a replay to a URL, and you could get back all kinds of information about the replay including basic info (like players, races, map name, duration, winners etc), the in-game chat of the players, and advanced info like the complete action list of players.
Or you could query the map preview image of of a replay based on the map file name info of a replay.

The service would not be free, a customer would have to pay based on usage (the number of API calls).

Benefits: you could operate a replay site without the need of a replay parser engine. Or you could use this service to validate your data. Or you might have your own replay parser engine, but you could use this to get advanced details about a game your engine is not capable of. This service would also get the computing load off of your server, and you could concentrate on your site's design and logic (and leave the replay parsing logic to me).
This would not require any technology on your server (like Java) as all API calls would be standard HTTP or HTTPS requests/responses. The responses would be likely XML or JSON (or plain text).

My question: Is there a need for a Parsing Service like this?
If you would be interested in using this Parsing service, please let me know by sending an email to me (my email is on the About dialog), and please describe in short what you would expect from a service like this.

June 27 - Sc2gears 6.0.3

The auto-updater was not launched properly on some computers with versions 6.0 and 6.0.1. It's fixed now.
The error is caused by folder names that contain spaces. So if you have Sc2gears in such a folder (or inside of a folder) that has spaces, here's how you can run the update properly:

Copy Sc2gears to a folder without spaces, for example "C:\Sc2gears". Start it, run the updater, after that you can copy it back.

The new version will not have this error and will run the auto-update properly.
